Introduction

Part of what makes forms complex for accessibility is the variety of feedback we need to give our users.

We can give live updates on how well they are meeting the instructions and requirements.

i.e. character counts in textareas, password strength, etc. This allows us to prevent errors before the user submits the form.

When the form is submitted, we must let the user know if there were any errors so they can be fixed before we send the data to the server.

We need to make sure that the info we're sending to the server is as clean as possible so it can be used there safely.

For accessibility we need to make sure that all of this feedback is available to everyone.

Live region states and properties

Live regions are what we use to give in the moment feedback. We have ARIA roles and attributes which can force a screen reader to make announcements. But with great power comes great responsibility.

Elements that are dynamically inserted after the initial page load can cause two potential problems for screen readers.

Why not just update character counts or insert search results using JavaScript? Becuase the way that screen reader software works can make it harder for it to notice these changes and inform the user.

Problem 1:

Screen readers “buffer” pages as they are loaded. Any content that is added after this time many not be picked up by the screen reader.

Some screen readers are improving on this, but there's still a long way to go. It's much easier for us to explicitly tell the software what we want it to know.

Problem 2:

Screen readers can only focus on one part of the page at a time. If something changes on another area of the page, screen readers will not announce this change unless the user moves to that area.

So we have to decide if we want to force them to get that update earlier. If you're not sure, have a chat with your UX or design team. Both of these problems can be solved by using an ARIA live region.

The live region attributes allow authors to notify screen readers when content is updated. The possible live region attributes are:

aria-live
aria-atomic
aria-relevant
aria-busy (state)

aria-live

There are three possible values:

  • off
  • polite
  • assertive

Off

Assistive Technologies will be aware that changes have occurred within the element. However, these changes should not be announced unless the region is currently in focus.

The off value should be used for information that is not critical, where changes do not need to be announced.

<div aria-live="off">
</div>

Polite

Assistive Technologies should announce updates at the next graceful opportunity.

The polite value can be used for warning notifications that users may need to know.

<div aria-live="polite">
</div>

Assertive

Assistive Technologies should announce updates immediately.

The assertive value should only be used if the interruption is imperative for users to know immediately.

<div aria-live="assertive">
</div>

aria-atomic

The aria-atomic attribute indicates whether assistive technologies will present all or only parts of the changed region.

The false attribute will present only the changed regions. This is the default.

<div
  aria-live="polite"
  aria-atomic="false"
>
</div>

The true attribute will present the region as a whole when changes are detected.

<div
  aria-live="polite"
  aria-atomic="true"
>
</div>

This attribute becomes important depending on the amount of content that needs to be announced.

<div
  aria-live="polite"
  aria-atomic="false"
>
  There are 3 items available.
  Use the arrow keys to navigate.
</div>
<div
  aria-live="polite"
  aria-atomic="true"
>
  There are 3 items available.
  Use the arrow keys to navigate.
</div>

The aria-live attribute should always be present when using the aria-atomic attribute.

<div
  aria-live="[ off | polite | assertive ]"
  aria-atomic=" [ false | true ]"
>
</div>

aria-relevant

The aria-relevant attribute defines what sorts of changes will be presented to screen readers.

The additions value informs Assistive Technologies of nodes that are dynamically inserted within the live region.

<div
  aria-live="polite"
  aria-relevant="additions"
>
</div>

The removals value informs Assistive Technologies of nodes that are dynamically deleted within the live region.

<div
  aria-live="polite"
  aria-relevant="removals"
>
</div>

The text value informs Assistive Technologies of dynamic changes to the textual content within existing nodes within the live region.

<div
  aria-live="polite"
  aria-relevant="text"
>
</div>

The all value informs Assistive Technologies of dynamic insertion or deletion of nodes within the live region.

<div
  aria-live="polite"
  aria-relevant="all"
>
</div>

The aria-live attribute should always be present when using the aria-relevant attribute.

<div
  aria-live="[ off | polite | assertive ]"
  aria-relevant=" [ additions | removals | text | all ]"
>
</div>

aria-busy

The aria-busy attribute indicates whether an element, and its subtree, are currently being updated.

The aria-busy attribute is a state, which means that it needs to be changed dynamically using JavaScript to determine the current state.

The value can be set to true when content is being loaded within the live region.

<div
  aria-live="polite"
  aria-busy="true"
>
</div>

The value can be set to false when the content loading has completed within the live region.

<div
  aria-live="polite"
  aria-busy="false"
>
</div>

The aria-live attribute should always be present when using the aria-busy attribute.

<div
  aria-live="[ off | polite | assertive ]"
  aria-busy=" [ true | false ]"
>
</div>

Live region roles

Live region roles can be used to define regions of a page that may be updated as a result of an external event.

All five live region roles have implicit live region properties:

Role Implicit aria-live Implicit aria-atomic
role="alert" "assertive" "true"
role="status" "polite" "true"
role="log" "polite" N/A
role="marquee" "off" N/A
role="timer" "off" N/A

This means that these implicit properties do not need to be defined when using the roles.

<div
  role="alert"
  aria-live="assertive"
  aria-atomic="true"
>
</div>
<div
  role="status"
  aria-live="polite"
  aria-atomic="true"
>
</div>
<div
  role="log"
  aria-live="polite"
>
</div>
<div
  role="marquee"
  aria-live="off"
>
</div>
<div
  role="timer"
  aria-live="off"
>
</div>

alert

The role="alert" is used for content that is important, and usually time-sensitive such as error messages.

As role="alert" is implictly set with aria-atomic="true", the entire region should be announced when changes are detected.

<div role="alert">
  Alert message information...
</div>

status

The role="status" is used for content that is advisory information that is not important enough to justify an alert.

As role="status" is implictly set with aria-atomic="true", the entire region should be announced when changes are detected.

<div role="status">
  Warning or success message information...
</div>

log

The role="log" is used to identify an element that creates a live region where new information is added in a meaningful order and old information may disappear.

<div role="log">
  Newly added log information...
</div>

marquee

The role="marquee" is used to identify where non-essential information changes frequently such as stock tickers and ad banners.

<div role="marquee">
  Newly added, non-essential banner information...
</div>

timer

The role="timer" is used to identify a numerical counter which indicates an amount of elapsed time from a start point, or the time remaining until an end point.

<div role="timer">
  Time information...
</div>

Password strength

A good example of a use case for aria-live is a password strength detector.

We may want to let the user know if their password is secure enough, so we present a strength meter.

It should dynmically update as soon as the user changes what they're typing.

This is a good opportunity to use an assertive live region.

Client-side form validation

Before submitting data to the server, it is important to ensure all required form controls are filled out, in the correct format. This is called client-side form validation.

Client-side form validation helps ensure data submitted matches the requirements in our instructions.

Why send the data on a round trip to the server if we already know it's not going to be what we need?

One key aspect of client side form validation is notifying users of errors in the form.

There are two fundamental approaches to error notification.

on blur

The first approach involves validating and notifying users of errors as focus leaves each individual form field, often referred to as onBlur.

As focus leaves each form field, users should be notified if the field is in a state of error.

On submit

The second approach involves validation and notifying users of errors as they attempt to submit the form, often referred to as onSubmit.

After the submit button is activated, users should be notified of all fields that are in a state of error.

In reality, forms that use onBlur for form field validation must still use onSubmit validation as well.

The reason is that users may not resolve individual form field errors before attempting to submit the form.

Why?

  • The user may have been focusing on specific fields and failed to notice the errors above.
  • Screen reader software may not have notified users of error messages.
  • Screen magnifier software may have magnified the page so that errors are no longer visible.

onBlur form field errors

From a developer perspective, the following should occur for fields that are in a state of error:

1. The error message must be programmatically associated with the invalid form field.

This can be done with matching aria-describedby and ID values.

<label for="name">Name</label>
<input
  type="text"
  id="name"
  aria-describedby="aaa"
>
<span id="aaa">Error message</span>

2. The invalid form field must be set with aria-invalid="true".

This will inform assistive technologies that the value entered into the input field does not conform to the format expected by the application.

<label for="name">Name</label>
<input
  type="text"
  id="name"
  aria-describedby="aaa"
  aria-invalid="true"
>
<span id="aaa">Error message</span>

Adding optional aria-live

The form field and error message can be wrapped inside a container that is set with aria-live.

<div aria-live="polite">
  <label for="name">Name</label>
  <input
    type="text"
    id="name"
    aria-describedby="aaa"
    aria-invalid="true"
  >
  <span id="aaa">Error message</span>
</div>

This means that the dynamically injected error state will be announced at some point after users leave the field.

The value of assertive should be avoided as the error message could be announced over other content.

<div aria-live="assertive">
  <label for="name">Name</label>
  <input
    type="text"
    id="name"
    aria-describedby="aaa"
    aria-invalid="true"
  >
  <span id="aaa">Error message</span>
</div>
<div aria-live="polite">
  <label for="name">Name</label>
  <input
    type="text"
    id="name"
    aria-describedby="aaa"
    aria-invalid="true"
  >
  <span id="aaa">Error message</span>
</div>

As assistive technologies become more advances, there is less need for aria-live properties.

Make sure to test this property before adding to determine if it is required.

onSubmit errors

If there any errors within the form when the user attempts to submit the form, users must be notified of all fields that are in an error state.

Focus could be sent to either:

  1. The first form field in a state of error.
  2. An error notification above the form.

Within your team, which of these two methods do you prefer?

Regardless, let's look at the second method in detail.

Steps when using an error notification above the form

There are many different ways to inject error notifications above forms.

This is one simple, robust solution that works well with all of the major screen readers.

1. The error notification must be placed above the form.

<div>
  <h4>
    Please fix the following two errors
  </h4>
  <ul>
    <li><a href="#">You must include...</a></li>
    <li><a href="#">You must use a...</a></li>
  </ul>
</div>
<form>
</form>

2. Focus should be sent to the heading inside the notification container.

<div>
  <h4>
    Please fix the following two errors
  </h4>
  <ul>
    <li><a href="#">You must include...</a></li>
    <li><a href="#">You must use a...</a></li>
  </ul>
</div>

3. The heading should be set with tabindex="-1".

This means that this heading will initially receive focus as soon as it is triggered, but then remain out of tabindex order afterwards.

<div>
  <h4 tabindex="-1">
    Please fix the following two errors
  </h4>
  <ul>
    <li><a href="#">You must include...</a></li>
    <li><a href="#">You must use a...</a></li>
  </ul>
</div>

4. The error notification should list all errors, allowing user to jump the relevant form errors.

<div>
  <h4 tabindex="-1">
    Please fix the following two errors
  </h4>
  <ul>
    <li><a href="#">You must include...</a></li>
    <li><a href="#">You must use a...</a></li>
  </ul>
</div>

Success messages

In some cases, you may need to inform users that the form has been submitted sucessfully.

Ideally, form-based success messages should follow many of the same principles as form-based error messages.

1. The success message must be presented to users.

The success message is normally positioned above the form or as a replacement for the entire form. If you're unsure, check with your UX team.

2. Focus must shift to the success message.

3. The success message must be set with role="status" as it is not an alert or error.

<div role="status">
  <h3>Sucess message heading alert</h3>
  <p>Success message content.</p>
</div>

4. The success message must also be set with tabindex="0" so that the element can receive focus as soon as it is triggered.

<div role="status" tabindex="0">
  <h3>Sucess message heading alert</h3>
  <p>Success message content.</p>
</div>

Recap

When we give our form notifications their correct roles and relationships, it's easy for the accessibility API to adapt our messages to the user's needs.

Make it clear what type of information you're giving, and how it relates to the form as a whole or to a specific input.
